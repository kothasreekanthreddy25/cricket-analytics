Part 1: TensorFlow.js Match Prediction Model

1.1 Architecture Overview

The match prediction system uses a feedforward neural network built with TensorFlow.js (@tensorflow/tfjs) that runs entirely server-side in the Node.js runtime. The model takes 14 normalized input features describing two competing teams and outputs win probabilities with a confidence score.

1.3 Neural Network Structure

The model is a Sequential neural network with the following layers:

LayerTypeUnitsActivationPurpose
1Dense32ReLUFeature extraction — learns non-linear combinations
2Dropout——20% dropout to prevent overfitting
3Dense16ReLUDeeper feature abstraction
4Dense2SoftmaxOutput: probabilities summing to 1.0

Model Construction Code

import * as tf from '@tensorflow/tfjs'

const FEATURE_COUNT = 14
const HIDDEN_UNITS_1 = 32
const HIDDEN_UNITS_2 = 16

function buildModel(): tf.LayersModel {
  const m = tf.sequential()

  // Layer 1: Dense with 32 units, ReLU activation
  m.add(tf.layers.dense({
    inputShape: [FEATURE_COUNT],
    units: HIDDEN_UNITS_1,
    activation: 'relu',
    kernelInitializer: 'glorotUniform',
  }))

  // Layer 2: Dropout (20%) for regularization
  m.add(tf.layers.dropout({ rate: 0.2 }))

  // Layer 3: Dense with 16 units, ReLU activation
  m.add(tf.layers.dense({
    units: HIDDEN_UNITS_2,
    activation: 'relu',
    kernelInitializer: 'glorotUniform',
  }))

  // Layer 4: Output layer with Softmax
  m.add(tf.layers.dense({
    units: 2,
    activation: 'softmax',
  }))

  // Compile with Adam optimizer
  m.compile({
    optimizer: tf.train.adam(0.001),
    loss: 'categoricalCrossentropy',
    metrics: ['accuracy'],
  })

  return m
}
Why Softmax Output? The softmax activation ensures the two output neurons produce valid probabilities that always sum to 1.0. If P(Team A) = 0.62, then P(Team B) = 0.38 automatically. This is more mathematically sound than independent sigmoid outputs.
Why Glorot Uniform Initialization? Glorot Uniform (Xavier) initialization scales weights based on the number of input/output neurons, preventing vanishing/exploding gradients during forward propagation. It's the standard for ReLU-activated networks.

1.4 Domain Weight Initialization

Since we don't have extensive historical training data, the model is initialized with domain-informed weights that emphasize the most predictive features based on cricket analytics knowledge: 

function initializeWithDomainWeights(m: tf.LayersModel) {
  const layers = m.layers
  const w1 = layers[0].getWeights()

  if (w1.length >= 2) {
    const weightShape = w1[0].shape
    const weightData = new Float32Array(weightShape[0] * weightShape[1])

    // Base: small random values centered on 0
    for (let i = 0; i < weightData.length; i++) {
      weightData[i] = (Math.random() - 0.5) * 0.3
    }

    // Boost weights for the most predictive features:
    // [0,1] = rankings, [2,3] = win rates, [4] = h2h, [8,9] = momentum
    const importantFeatures = [0, 1, 2, 3, 4, 8, 9]
    for (const feat of importantFeatures) {
      for (let j = 0; j < weightShape[1]; j++) {
        weightData[feat * weightShape[1] + j] *= 2.0  // 2x weight boost
      }
    }

    layers[0].setWeights([
      tf.tensor2d(weightData, weightShape),

    ])
  }
}
Feature Importance (Boosted 2x) Rankings, win rates, head-to-head, and momentum are the strongest predictors of match outcomes in cricket. Pitch type, format, and batting/bowling strength serve as contextual modifiers.

1.5 Prediction & Confidence Scoring

The predict() function converts features into a tensor, runs inference, and returns formatted probabilities with a confidence level: 

export function predict(features: MatchFeatures): PredictionResult {
  const m = getModel()
  const input = featuresToTensor(features)
  const output = m.predict(input) as tf.Tensor
  const probs = output.dataSync()  // [P(A), P(B)]

  // Memory cleanup
  input.dispose()
  output.dispose()

  // Round to 1 decimal place
  const teamAWinProb = Math.round(probs[0] * 1000) / 10
  const teamBWinProb = Math.round(probs[1] * 1000) / 10

  // Confidence based on probability gap
  const gap = Math.abs(probs[0] - probs[1])
  let confidence: 'high' | 'medium' | 'low'
  if (gap > 0.3) confidence = 'high'
  else if (gap > 0.15) confidence = 'medium'
  else confidence = 'low'

  return { teamAWinProb, teamBWinProb, confidence }
}

1.6 Training Pipeline

The model supports online training with historical match data. When training data becomes available, the trainModel() function fine-tunes the pre-initialized weights: 

export async function trainModel(
  trainingData: { features: MatchFeatures; winner: 'A' | 'B' }[]
): Promise<tf.History> {
  const m = getModel()

  // Convert features to 2D tensor [numSamples, 14]
  const xs = tf.tensor2d(trainingData.map(d => [
    d.features.teamARanking,
    d.features.teamBRanking,
    // ... all 14 features
  ]))

  // One-hot encode labels: A wins = [1,0], B wins = [0,1]
  const ys = tf.tensor2d(trainingData.map(d =>
    d.winner === 'A' ? [1, 0] : [0, 1]
  ))

  const history = await m.fit(xs, ys, {
    epochs: 50,
    batchSize: 16,
    validationSplit: 0.2,   // 20% held out for validation
    shuffle: true,
  })

  xs.dispose()
  ys.dispose()
  return history
}

Part 2: Socket.IO Real-Time Live Scores

2.1 Architecture Overview

The live score system uses Socket.IO to push real-time match updates to connected clients. The server polls the Roanuz Cricket API every 15 seconds for live match data and broadcasts score changes via WebSocket.

┌─────────────────────────────────────────────────────────────┐ │ Next.js Server │ │ │ │ pages/api/socket.ts │ │ ┌───────────────────┐ ┌──────────────────────┐ │ │ │ Socket.IO Server │────│ Live Score Poller │ │ │ │ (WebSocket) │ │ (15s interval) │ │ │ └────────┬──────────┘ └──────────┬───────────┘ │ │ │ │ │ │ │ score:update │ Roanuz API │ │ │ events │ GET /match/:key/ │ │ │ │ │ └───────────┼───────────────────────────┼─────────────────────┘ │ │ ▼ ▼ ┌───────────────────┐ ┌──────────────────────┐ │ Browser Client │ │ Roanuz Cricket API │ │ useLiveScores() │ │ api.sports.roanuz.com│ │ hook updates │ └──────────────────────┘ │ React state │ │ ──> Match Cards │ └───────────────────┘
Why Socket.IO over plain WebSocket? Socket.IO provides automatic reconnection, transport fallback (long-polling to WebSocket upgrade), room/namespace support, and typed events. It gracefully handles network drops which are common on mobile devices during live match viewing.

2.3 Server-Side: Socket.IO Bootstrap

Socket.IO attaches to the Next.js HTTP server via a Pages API route. This coexists with the App Router without conflicts. pages/api/socket.ts 

import { Server as SocketIOServer } from 'socket.io'
import type { NextApiRequest, NextApiResponse } from 'next'
import {
  setIO, incrementSubscribers,
  decrementSubscribers, getCachedScores,
} from '@/lib/live-score-poller'

export default function handler(req, res) {
  // Idempotent: only initialize once (survives hot reload)
  if (res.socket.server.io) {
    res.end()
    return
  }

  // Attach Socket.IO to the underlying Node HTTP server
  const io = new SocketIOServer(res.socket.server, {
    path: '/api/socket',
    addTrailingSlash: false,
    cors: {
      origin: ['http://localhost:3000'],
      methods: ['GET', 'POST'],
    },
    transports: ['polling', 'websocket'],
  })

  res.socket.server.io = io  // Persist on server object
  setIO(io)                  // Give poller broadcast access

  io.on('connection', (socket) => {
    socket.on('subscribe:live', () => {
      incrementSubscribers()

      // Send cached scores immediately (no wait for next poll)
      const cached = getCachedScores()
      if (cached.length > 0) {
        socket.emit('score:update', cached)
      }
    })

    socket.on('disconnect', () => {
      decrementSubscribers()
    })
  })

  res.end()
}
Key Design: Idempotent Initialization The res.socket.server.io check ensures Socket.IO is only initialized once, even when Next.js hot-reloads the API route during development. The io instance persists on the Node HTTP server object across module reloads.

2.4 Server-Side: Live Score Poller

A singleton service that polls Roanuz every 15 seconds. It only runs when clients are connected (subscriber count > 0), preventing unnecessary API calls. lib/live-score-poller.ts 

const POLL_INTERVAL_MS = 15_000  // 15 seconds

let pollingTimer: NodeJS.Timeout | null = null
let subscriberCount = 0

// Only start polling when clients are connected
function maybeStartPolling() {
  if (pollingTimer) return         // Already polling
  if (subscriberCount <= 0) return  // No clients
  pollOnce()                       // Immediate first poll
  pollingTimer = setInterval(pollOnce, POLL_INTERVAL_MS)
}

// Stop when all clients disconnect
function maybeStopPolling() {
  if (subscriberCount > 0) return
  if (pollingTimer) {
    clearInterval(pollingTimer)
    pollingTimer = null
  }
}

Score Extraction from Roanuz API

The Roanuz API returns innings data in a nested structure. The score field is an object (not a number), and overs is an array: 

// Roanuz API innings structure:
// innings: { "a_1": { score_str: "185/4 in 16.3", score: { runs: 185, balls: 99 },
//                       wickets: 4, overs: [16, 3] } }

function extractScore(matchData): { scoreA: string; scoreB: string } {
  const innings = matchData.play.innings
  const teamKeys = Object.keys(matchData.teams)

  for (const ik of Object.keys(innings)) {
    const inn = innings[ik]

    // Prefer score_str ("185/4 in 16.3") if available
    let scoreStr = inn?.score_str || ''

    // Fallback: build from nested fields
    if (!scoreStr && inn?.score) {
      const runs = typeof inn.score === 'object'
        ? inn.score.runs    // { runs: 185, balls: 99 }
        : inn.score
      const wickets = inn.wickets
      const overs = Array.isArray(inn.overs)
        ? `${inn.overs[0]}.${inn.overs[1]}`  // [16, 3] -> "16.3"
        : inn.overs
      scoreStr = `${runs}/${wickets} (${overs} ov)`
    }

    // Innings key "a_1" -> team "a", "b_1" -> team "b"
    const teamChar = ik.split('_')[0]
    if (teamChar === teamKeys[0]) scoreA = scoreStr
    else scoreB = scoreStr
  }
}

}

2.5 Client-Side: React Hook

The useLiveScores() hook manages the Socket.IO connection lifecycle and exposes reactive state. lib/use-live-scores.ts 

'use client'

import { useEffect, useState, useRef, useCallback } from 'react'
import { io, Socket } from 'socket.io-client'

export function useLiveScores() {
  const [scores, setScores] = useState<Map<string, LiveScoreUpdate>>(new Map())
  const [connected, setConnected] = useState(false)

  useEffect(() => {
    // Step 1: Bootstrap Socket.IO server via HTTP request
    fetch('/api/socket').then(() => {

      // Step 2: Connect via WebSocket
      const socket = io({
        path: '/api/socket',
        transports: ['polling', 'websocket'],
        reconnection: true,
        reconnectionAttempts: Infinity,
        reconnectionDelay: 1000,
        reconnectionDelayMax: 10000,
      })

      // Step 3: Subscribe to live scores on connect
      socket.on('connect', () => {
        setConnected(true)
        socket.emit('subscribe:live')
      })

      // Step 4: Handle incoming score updates
      socket.on('score:update', (updates) => {
        setScores((prev) => {
          const next = new Map(prev)
          for (const update of updates) {
            next.set(update.matchKey, update)
          }
          return next  // O(1) lookup by match key
        })
      })

      socket.on('disconnect', () => setConnected(false))
    })

    // Cleanup on unmount
    return () => {
      socket.emit('unsubscribe:live')
      socket.disconnect()
    }
  }, [])

  return { scores, connected, getScore: (key) => scores.get(key) }
}
Connection Flow
  1. fetch('/api/socket') triggers the Pages API handler, initializing Socket.IO on the server
  2. io({ path: '/api/socket' }) establishes the WebSocket connection
  3. On connect, client emits subscribe:live — server starts polling Roanuz
  4. Every 15s, server broadcasts score:update with latest live scores
  5. On unmount, client emits unsubscribe:live — server stops polling when no clients remain

2.6 UI Integration

The analysis page merges Socket.IO score updates into the match card state via a useEffect: app/analysis/page.tsx 

import { useLiveScores } from '@/lib/use-live-scores'

export default function AnalysisPage() {
  const [matches, setMatches] = useState([])
  const { scores: liveScores, connected: socketConnected } = useLiveScores()

  // Merge live scores into match cards whenever updates arrive
  useEffect(() => {
    if (liveScores.size === 0) return
    setMatches((prev) =>
      prev.map((m) => {
        const live = liveScores.get(m.key)
        if (!live) return m
        return {
          ...m,
          scoreA: live.scoreA || m.scoreA,
          scoreB: live.scoreB || m.scoreB,
          statusNote: live.statusNote || m.statusNote,
          status: live.status || m.status,
        }
      })
    )
  }, [liveScores])

  // Connection indicator in header
  return (
    <div>
      {socketConnected && (
        <span className="badge green">
          <span className="pulse-dot" />
          Live Updates
        </span>
      )}
      ...
    </div>
  )
}
Automatic Card Updates The MatchCard component renders from the matches state array. When liveScores changes (via Socket.IO), the useEffect merges new data into the state, triggering a React re-render. Cards update seamlessly without page refresh.
